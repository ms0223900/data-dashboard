雖然檔案格式（Markdown）不變，但在 **內容比重** 與 **規範深度** 上會有顯著差異。以下是針對這兩類專案的設定建議：

---

### 1. 快速做完型 (MVP / 實驗性專案)
這類專案的目標是**速度**。規則應聚焦於「減少廢話」與「快速生成功能」。

*   **AGENTS.md / .cursorrules 重點：**
    *   **忽略最佳實踐**：明確告訴 AI 不需要拆分太細的元件或過度的抽象化。
    *   **單一檔案原則**：傾向於將邏輯寫在同一個檔案內，減少跨檔案跳轉。
    *   **預設樣式**：直接指定常用的 UI 庫（如 Tailwind CSS），讓 AI 直接給出含樣式的代碼。

**範例片段：**
```markdown
# 快速開發規則
- 優先使用 Inline Styles (Tailwind)，不需要另建 CSS 檔案。
- 邏輯與 UI 盡量放在同一個檔案，減少複雜的資料夾層級。
- 註解只需說明「為什麼這麼做」，不需詳細的文件化。
- 只要能運作，不需追求完美的效能優化。
```

---

### 2. 長期維護型 (正式產品 / 團隊協作)
這類專案的目標是**可讀性**與**穩定性**。規則應聚焦於「架構約束」與「自動化測試」。

*   **AGENTS.md / .cursorrules 重點：**
    *   **嚴謹的設計模式**：如 Clean Architecture 或特定 Folder Structure。
    *   **型別安全**：強制要求 TypeScript 的型別定義，禁止使用 `any`。
    *   **文檔化要求**：要求每個 Function 必須包含 JSDoc。
    *   **測試驅動**：要求修改代碼時同步更新測試案例。

**範例片段：**
```markdown
# 長期維護規則
- 強制執行單一功能原則 (Single Responsibility Principle)。
- 所有元件必須存存放於 `/components` 下的對應子目錄。
- 必須撰寫 TypeScript Interface，嚴禁使用 any。
- 每次新增 API 邏輯時，必須同步更新 `/docs/api.md`。
```

---

### 兩種模式的對比表

| 特性 | 快速做完型 (MVP) | 長期維護型 (Scale) |
| :--- | :--- | :--- |
| **程式碼密度** | 高 (單一檔案多邏輯) | 低 (模組化、拆分檔案) |
| **型別要求** | 寬鬆 (可接受推斷型別) | 嚴格 (需定義 Interface/Type) |
| **註解風格** | 簡略、功能導向 | 詳細、符合文檔規範 (JSDoc) |
| **錯誤處理** | 基本 (能動就好) | 完整 (包含 Error Boundary) |

---

### 如何導入
1.  **快速型**：將規則直接寫在專案根目錄的 `.cursorrules`，讓 AI 在每次對話都能讀取到「速度優先」的指令。
2.  **長期型**：建議利用 `.cursor/rules/` 資料夾。你可以將規則拆分成 `typescript.md`, `testing.md`, `ui-style.md`，這樣當你在開發特定部分時，AI 能根據檔案內容精準套用規則。
