# User Stories 拆解

## Description

使用者故事 - 將需求拆解為個別小單元的實作任務

## Usage

將需求拆解為個別小單元的實作任務，每個任務都有明確的輸入輸出、驗收條件和依賴關係

## Prompt Template

```
請你幫我根據以下脈絡，整理為個別小單元的實作任務：

- 需求： @{需求文件}
- 模組拆解：@{模組拆解文件}
- 解決方案： @{解決方案文件}

任務請依照以下原則來生成，每一項任務不要太大：

- 每個任務都要能夠獨立完成，需要有明確的：
  - **輸入格式**：需要什麼樣的輸入資料？
  - **輸出格式**：會產生什麼樣的輸出結果？
  - **驗收條件**：什麼情況下才算完成？
  - **測試策略**：每個功能／實作任務標註 Test-First / Test-After / Exploratory（判準見下方「測試策略判斷」）；純 Test-First 測試準備任務則標註其預期紅燈完成條件
  - **依賴關係**：需要先完成哪些任務？
- 任務之間的依賴關係要明確，可以參考模組拆解的依賴關係。
- 若某個任務判定為 Test-First，且屬於可獨立驗證的邏輯（非 UI 排版類），優先拆成「測試」與「實作」兩個獨立任務，依賴關係設為「實作任務依賴測試任務」，而不是把兩者混在同一個任務描述裡。測試任務須明載完成條件為「聚焦測試因功能尚未實作而預期紅燈」；實作任務才標記為 **測試策略：Test-First**，並負責將該測試轉綠及驗收功能 AC。

每個任務請幫我存成個別獨立的文件，文件名稱為「{任務名稱}.md」，方便後續追蹤和管理。
```

## 測試策略判斷（Test-First / Test-After / Exploratory）

拆任務時，每個任務都要判斷「測試該先寫還是後補」，這個判斷會直接影響 `next-task` 之後的排程與分派方式，所以要在拆 US 的當下就決定，而不是留到實作時才臨場決定。

**目前不強制整個專案走 TDD**，但下列情境建議預設用 **Test-First**（先寫會紅的測試，待實作完成後轉綠）：

| 情境 | 建議測試策略 | 理由 |
|---|---|---|
| **新功能**，且邏輯明確可轉成具體 input/output（計算、驗證規則、狀態轉換、API 資料轉換） | **Test-First** | 先紅後綠的節奏，天然適合後續自動化 loop-engineering：agent 只要對著失敗測試反覆迭代，測試轉綠就是明確的完成訊號 |
| **修改既有邏輯**（含 bug 修復，已能明確重現） | **Test-First** | 先寫一支會重現問題的失敗測試，鎖住預期行為，實作/修正到轉綠，同時避免回歸 |
| UI 排版/樣式/文案調整 | **Test-After**（或不寫自動化測試） | 視覺回饋比預先寫斷言更有效率；若判定不寫測試，需在任務內說明原因 |
| 規格仍在探索、可能大幅變動（spike／雛形） | **Exploratory**（可不寫測試） | 先出雛形讓 PM/設計確認，定案後再視情況補測試；同樣需要說明原因 |

寫入任務文件時的格式：

```
**測試策略**：Test-First
> 理由：邏輯明確（下注金額驗證規則），適合先寫失敗測試再實作。
```

若判定為 Test-After 或 Exploratory，理由同樣要寫清楚，避免後續執行任務的人（或 `next-task`／`adjust`／`feature`）無法判斷「為什麼這裡不用先寫測試」。

**下游如何使用這個欄位**：`next-task` 分派任務時會讀取此欄位，決定分派順序與指令內容；`feature`／`adjust` 實作時會依此欄位決定先寫測試還是先實作。

## User Story Components

- **標準格式**：「作為...我想要...以便...」
- **驗收條件**（Checklist）
- **測試策略**：Test-First / Test-After / Exploratory + 一行理由（見上方「測試策略判斷」）
- **優先級標記**
- **相關功能對應**

## 精簡原則

US 常會被 `adjust` 多次追加/調整，經過幾輪修改後容易越寫越長。生成或調整時：

- 每個欄位只寫一次該講的事，不要在「作為/我想要/以便」的敘述裡把驗收條件的內容又重述一遍。
- 驗收條件用具體、可測試的短句，不要展開成一段話。
- 若發現既有 US 已經寫得很囉唆，可提醒使用者之後用 `/doc-trim` 回頭精簡措辭；本 skill 本身只負責產出時就寫精簡，不主動回頭改寫既有檔案的表達方式。

## Example

```
### US-002：查看「我們的日常」文章列表
**作為** 求職者
**我想要** 瀏覽「我們的日常」文章列表
**以便** 了解公司內部日常活動與文化

**驗收條件**：
- [ ] 區塊標題顯示「我們的日常」與「Career News」
- [ ] 每篇文章顯示標題、發布日期、標籤徽章
- [ ] 每篇文章前有藍色箭頭圖示，表示可點擊
- [ ] 文章列表垂直排列，每篇文章間有適當間距
- [ ] 預設每頁顯示 3 篇文章
- [ ] 標題可點擊，連結至文章詳情頁
- [ ] 日期格式為 YYYY-MM-DD
- [ ] 響應式設計在各種裝置上正常運作

**測試策略**：Test-After
> 理由：屬於清單排版/顯示調整，視覺回饋比預先寫斷言更有效率。

**優先級**：P0
**相關功能**：文章列表區塊（2.1.2）
```
